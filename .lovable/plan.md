
Цель: перенести сайт с текущего встроенного backend на ваш собственный Supabase-проект, сохранив контент, админку, роли и медиафайлы.

1. Подготовить целевой Supabase-проект
- Вы создаёте свой Supabase-проект в своём аккаунте.
- Я подключу этот проект к приложению как внешний backend через интеграцию.
- Важно: текущий Lovable Cloud у этого проекта уже включён, поэтому “отвязать” его полностью нельзя. Корректный путь — переключить приложение на ваш Supabase и перенести данные/схему, либо сделать это в копии проекта.

2. Перенести структуру базы данных
- Воссоздать в вашем Supabase текущие сущности:
  - `services`
  - `doctors`
  - `news`
  - `legal_acts`
  - `user_roles`
  - enum `app_role`
  - функции `has_role()` и `is_admin()`
- Воссоздать политики доступа:
  - публичное чтение активного контента
  - CRUD только для администраторов
- Проверить, что структура совпадает с текущими типами и запросами фронтенда.

3. Перенести аутентификацию и роли
- Настроить email/password auth в вашем проекте.
- Создать администратора в вашем Supabase Auth.
- Назначить ему роль `admin` через таблицу `user_roles`.
- Проверить, что текущая логика `AuthContext` и защита `/admin` продолжают работать без переписывания бизнес-логики.

4. Перенести файлы из storage
- Создать bucket `media` как public.
- Скопировать туда изображения и другие медиа из текущего bucket `media`.
- Проверить, что `MediaAdmin` умеет:
  - загружать файлы
  - удалять файлы
  - получать public URL
- При необходимости обновить ссылки на файлы в контентных таблицах.

5. Переключить приложение на новый Supabase
- Обновить привязку проекта к вашему Supabase так, чтобы автогенерируемые файлы использовали уже ваш backend:
  - `src/integrations/supabase/client.ts`
  - `src/integrations/supabase/types.ts`
  - `supabase/config.toml`
- Я не буду редактировать эти файлы вручную; они должны обновиться через интеграцию.
- После переключения сверю все места, где используется клиент Supabase (контент, поиск, админка, storage, auth).

6. Проверить весь пользовательский функционал
- Публичная часть:
  - услуги
  - врачи
  - новости
  - НПА
  - поиск
- Админка:
  - вход
  - создание/редактирование/удаление записей
  - загрузка медиа
- Убедиться, что RLS не ломает публичные страницы и админский доступ.

7. Очистить привязки к старому backend
- Проверить, нет ли жёстких ссылок на старый project id / URL.
- Убедиться, что приложение полностью читает и пишет данные уже в ваш Supabase.
- При необходимости обновить README/инструкции по дальнейшему сопровождению.

- Если прямое переключение окажется ограничено из-за уже включённого Lovable Cloud, я выполню безопасный вариант: подготовлю миграцию в копию проекта, привязанную к вашему Supabase, и перенесу туда код и данные.

Технические детали
- Автогенерируемые файлы Supabase нельзя править вручную; переключение должно идти через подключение интеграции.
- Политики доступа будут восстановлены на уровне БД, а не на клиенте.
- Админские роли останутся в отдельной таблице `user_roles`, что безопасно и совместимо с текущим кодом.
- Storage bucket `media` должен быть public, иначе текущий `getPublicUrl()` и превью в админке сломаются.
- После подключения нового Supabase нужно будет заново сгенерировать типы БД через интеграцию, чтобы фронтенд остался типобезопасным.

После вашего одобрения я перейду к выполнению: подготовлю перенос схемы, помогу подключить ваш Supabase, затем переключу приложение и проверю весь сайт end-to-end.
